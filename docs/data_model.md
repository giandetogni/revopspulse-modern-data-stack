# Data Model

RevOpsPulse follows a layered analytics engineering architecture designed to separate raw ingestion, light standardization, business logic and analytics-ready reporting models.

## Layer Overview

| Layer | Purpose | Examples |
|---|---|---|
| RAW | Preserve extracted data with ingestion metadata and minimal transformation | `raw.json_records`, raw CSV/JSONL files |
| STAGING | Standardize names, types and light cleaning | `stg_postgres__payments`, `stg_api__opportunities`, `stg_events__product_events` |
| INTERMEDIATE | Apply reusable business logic | `int_payments_enriched`, `int_opportunities_enriched`, `int_product_usage_account_daily` |
| MARTS | Serve analytics-ready facts, dimensions and metric tables | `fact_payments`, `mart_revenue_monthly`, `mart_sales_commissions_monthly` |
| SNAPSHOTS | Track slowly changing records over time | `subscription_snapshot` |

## RAW Layer

The RAW layer stores extracted records with ingestion metadata. The project currently uses local raw files and a Postgres raw table for JSON records.

### Raw Files

Postgres extracts are written as CSV files under:

`raw/postgres/<entity>/load_date=<date>/batch_id=<batch_id>/data.csv`

API extracts are written as JSONL files under:

`raw/api/<endpoint>/load_date=<date>/batch_id=<batch_id>/data.jsonl`

Product events are written as JSONL files under:

`raw/events/product_events/load_date=<date>/batch_id=<batch_id>/data.jsonl`

### Raw Metadata

Raw records include metadata such as:

| Column | Description |
|---|---|
| `source_system` | Source system name, such as `postgres`, `mock_crm_api` or `product_events` |
| `source_entity` | Source entity or endpoint |
| `source_table_or_endpoint` | Source table or API endpoint |
| `batch_id` | Unique extraction batch identifier |
| `extracted_at` | Extraction timestamp |
| `load_date` | Logical load date |
| `raw_file_path` | Path to the raw file generated by the extractor |
| `record_hash` | Hash used for deduplication and lineage |

### Raw JSON Table

API and product event JSON records are loaded into:

`raw.json_records`

This table stores source metadata and JSON payloads used by dbt staging models.

## INTERMEDIATE Layer

Intermediate models centralize reusable business logic so marts remain simpler and easier to validate.

| Model | Purpose |
|---|---|
| `int_payments_enriched` | Combines payments, invoices and refunds to calculate net paid revenue and refund flags |
| `int_opportunities_enriched` | Enriches CRM opportunities with sales rep and campaign context |
| `int_product_usage_account_daily` | Aggregates product events by account and day |

## MARTS Layer

The marts layer contains facts, dimensions and final metric models.

## Dimensions

| Model | Grain | Description |
|---|---|---|
| `dim_accounts` | One row per account | Account attributes such as industry, company size and country |
| `dim_customers` | One row per customer | Customer identity and account relationship |
| `dim_sales_reps` | One row per sales rep | Sales rep team, region and active status |

## Facts

| Model | Grain | Description |
|---|---|---|
| `fact_payments` | One row per payment | Payment amount, refund amount, net paid amount and invoice status |
| `fact_opportunities` | One row per opportunity | CRM opportunity amount, stage, sales rep and campaign attribution |
| `fact_product_events` | One row per product event | Incremental event fact keyed by `event_id` |

## Final Metric Marts

| Model | Grain | Description |
|---|---|---|
| `mart_revenue_monthly` | One row per month | Revenue, MRR, ARR, churn, retention and payment quality metrics |
| `mart_product_usage_daily` | One row per day | Product usage, activation, adoption and cancellation intent metrics |
| `mart_marketing_roi_monthly` | One row per campaign/month | Spend, funnel conversion, CAC and marketing ROI |
| `mart_sales_commissions_monthly` | One row per sales rep/month | Quota attainment, commissionable revenue, payout and clawbacks |
| `mart_data_quality_summary` | One row per quality check/run | Data quality status, failed records, failure rate and business impact |

## Incremental Model

`fact_product_events` is materialized incrementally and keyed by `event_id`.

The model only processes product events with `event_timestamp` greater than the maximum event timestamp already present in the target table.

This demonstrates an incremental loading pattern for append-oriented event data.

## Snapshot Model

`subscription_snapshot` tracks changes to subscriptions using dbt snapshots.

| Snapshot Setting | Value |
|---|---|
| Unique key | `subscription_id` |
| Strategy | `timestamp` |
| Updated-at column | `updated_at` |
| Target schema | `snapshots` |

This provides a foundation for SCD Type 2 analysis of subscription lifecycle changes.

## Business Logic Summary

### Revenue

Revenue marts use successful paid invoices and net paid amounts after refunds. MRR is estimated from active and past-due subscriptions.

### Product Usage

Product usage marts aggregate event data into daily account-level and portfolio-level usage metrics.

### Marketing ROI

Marketing marts connect campaign spend, leads and opportunities to calculate funnel conversion, CAC and ROI.

### Sales Commissions

Commission marts attribute paid revenue to sales reps through account-level closed-won opportunities.

Commission rules:

- closed-won opportunity plus paid invoice generates eligibility;
- base commission is 5% of net paid revenue;
- reps at or above 100% quota earn 8% on above-quota revenue;
- refunds within 30 days create clawbacks;
- failed payments, unpaid invoices and non-closed-won opportunities are not commissionable.

## Data Quality Coverage

The model includes dbt generic tests, custom business rule tests and a data quality mart.

Examples:

| Test | Business Rule |
|---|---|
| `amount_non_negative` | Monetary amounts should not be negative unless intentionally modeled |
| `valid_subscription_period` | Subscription end date must not be before start date |
| `refund_not_greater_than_payment` | Refund amount should not exceed payment amount |
| `event_timestamp_not_future` | Product events should not occur in the future |
| `commission_requires_paid_invoice` | Commission requires a paid invoice and eligible opportunity logic |

## Known Modeling Limitations

MRR is calculated from current subscription status and plan price. A production implementation should use historical subscription snapshots for month-accurate MRR.

Marketing attribution is simplified at campaign/month level. Leads and opportunities can occur in different months, so monthly conversion rates can exceed 1.0 in synthetic data.

Sales commission attribution uses one sales rep per account based on matched opportunity data. Production-grade compensation systems may require split credit, ramping quotas, compensation plan versions and territory rules.

The project uses synthetic data only. It is not connected to any real company or internal system.
